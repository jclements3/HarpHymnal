Goal

A Python tool that models a diatonic jazz harpist preparing Sunday prelude improvisations on a 47-string lever harp. Not a real-time performer — a practice-week exploration assistant. Across the week, the tool generates MIDI + score candidates. The harpist auditions, curates, and drills favorite fragments. On Sunday, the harpist improvises by recombining drilled fragments in real time.

Physical constraints

  • 47 strings, diatonic, lever-operated (levers raise a string by a semitone).
  • Player has 8 fingers (4 per hand, no pinkies).
  • Each hand spans ≤ 10 strings (≈ a diatonic 10th).
  • Lever flips during performance are expensive: occasional single-bar flip-and-back is OK, sustained flips are OK, frequent flipping is not.
  • Shapes are pre-curated; the tool selects from them, it does not invent.
  • String damping and sympathetic ringing matter to texture.

Structural biases

Density axis

prefer 1-2 attacks per beat; reserve 3-4 attacks for arrivals (phrase ends, cadences, climaxes)

penalty target: high density in non-arrival bars

Vertical spread axis

prefer 3-5 octave spread; avoid SATB-like 2-octave middle-register cluster

penalty target: narrow spread in middle register for sustained passages

SATB-zone avoidance

penalize combinations with high density + narrow spread + middle register + on-the-beat block attacks

penalty target: combined cluster of these four

Expose the extremes

the audience tracks the highest and the lowest voice — keep both clear. Prefer voicings where the top (melody) and the bottom (bass) are registrally isolated from the middle; put color in the middle instead of close-position chord tones directly under the melody

penalty target: (a) melody + plus_1/plus_2 cluster just below the top; (b) bass crowded into mid-register fill rather than sitting low; (c) no LH strike below string 15 (C3) for the entire hymn

Decisions (resolved scope flags)

  1. Shape library — optimize for 47-string. Extend pretty_fraction.py: C1..G7, STRING_MIN=1 at C1, STRING_MAX=47 at G7, seven octaves. ≤10-string hand span; gap between hands is flexible.
  2. v1 activity / texture tactics — full pool is selectable. No walking-skeleton subset; build the foundation so every tactic in tactics.json is renderable by the MIDI + score pipeline.
  3. Fragment length — phrase-aligned, using the phrase boundaries already computed by the parser (data/hymns/<slug>.json → phrases[]).
  4. Shape count — "300+" stands as the output expectation of the 47-string extension across qualities × registers.

Open questions

Still unresolved; tracked in REHARM_TACTICS.md:

  1. Verse differentiation — identical per verse, or varied (register/texture intensification)?
  2. Intro / outro handling — separate artifacts or a phrase_role value?
  3. Quotation / allusion — in scope?
  4. Session-level memory across multiple preludes in one service.
  5. Liturgical mood parameter — Lent restrained, Easter celebratory.
  6. Per-hymn deny-list for inappropriate treatments.

Process

  1. Edit prose in REHARM_TACTICS.md for intent / rationale changes.
  2. Edit data/reharm/tactics.json for adding / removing / modifying tactic records or compatibility edges.
  3. Regenerate: python3 trefoil/build_reharm_tactics.py.
  4. Commit MD + JSON + HTML together.

A tactic that doesn't survive contact with real hymn data gets redefined, flagged rare, or removed — with the reason recorded in its record's note field.
